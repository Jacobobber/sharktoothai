Security

Sharktooth is designed to meet the security expectations of modern dealership IT environments. The following outlines how data is handled, protected, and accessed within the platform.

Data handling

Sharktooth processes structured repair order data for analytical purposes. The platform is designed to minimize data exposure and limit processing to what is necessary for analysis.

  • Personally identifiable information is deterministically redacted prior to analysis
  • Customer names, addresses, phone numbers, and VINs are not stored within the AI layer
  • Free-form text fields are not required for core analysis
  • Data is handled in a read-only analytical context

Access control

Access to Sharktooth is restricted to authorized users and governed by role-based controls.

  • Role-based access control (RBAC) is enforced
  • Access events are logged
  • Administrative actions are restricted and auditable
  • Authentication is required for all application access

Infrastructure and isolation

Sharktooth is deployed within isolated infrastructure designed to limit lateral access and reduce blast radius.

  • Application components are logically isolated
  • Data processing is separated from public-facing services
  • Network access is restricted to required paths only
  • Production systems are not accessible from public endpoints beyond the application interface

Operational security

Operational controls are in place to support system integrity and traceability.

  • Changes are managed through controlled deployment processes
  • Errors and system events are logged
  • Public endpoints are rate-limited
  • Security-relevant events are reviewable

Compliance posture

Sharktooth is designed to align with common dealership and enterprise security expectations.

  • Least-privilege access principles are applied
  • Data retention is limited to operational necessity
  • The platform avoids direct integration paths that would require write access to dealer systems
